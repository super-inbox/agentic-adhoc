# Progress update — Visual Intent Routing eval

**Date:** 2026-07-01 · **Author:** Rong · **Phase:** 1 (Benchmark) — Layer-1 Template Routing Accuracy

## TL;DR

Layer-1 routing eval runs end-to-end: **capability KB → 58-query gold → Path A / B / union scorers**, plus a new **ontology-enriched gold** that gives per-axis (Subject × Info-type × Layout) error attribution. **The one blocker is the gate: the gold is still a Claude draft — it needs human sign-off before any comparative number (esp. KB-lift) is trustworthy.** A decision-only review checklist is ready to make that fast.

```
   [built] capability KB ──► [built] routing gold (58q) ──► [built] Path A/B/union scores
                                     │
                                     ├─► [built] ontology-enriched gold (per-axis attribution)
                                     │
                                     └─► [GATE] human review of gold  ◄── we are here
                                             │
                                             └─► [next] trustworthy KB-lift + cell-level scoring
```

## 1. What v1 delivers

| Piece | State | Detail |
|---|---|---|
| **Capability KB** | ✅ built | 227 generatable templates, 2,804 examples attributed. Mines each template's *real* generative range from its examples (param values, tags, aliases), not its name/description. |
| **Routing gold** | ⚠️ draft | 58 queries → `acceptable_template_ids[]` (a SET, for multi-valid), `primary`, `ambiguity`, `near_miss`, rationale+evidence. |
| **Ontology-enriched gold** | ✅ built | Every template tagged with its `(Subject × Info-type × Layout)` cell → routing errors attributable to a specific axis. |
| **Scorers** | ✅ built | Path A (keyword/alias retrieval, offline) · Path B (live gpt-4o-mini, exact production prompt) · union. Top-1/3 + hit@any, bucketed by ambiguity. |

Gold composition: ambiguity low 11 / med 37 / high 10 · locale en 37 / zh 21 · 7 content-gap queries (`acceptable=[]`).

**Anti-contamination guardrail:** labels were NOT produced by running Path A or B (the systems under test) — so the eval doesn't grade the matcher against its own output.

## 2. Results — Path A vs Path B vs union

| | top-1 | top-3 | hit@any | low | medium | high | gap-clean |
|---|---|---|---|---|---|---|---|
| **Path A** (retrieval) | **53%** | 78% | 86% | 55% | 43% | **80%** | 29% |
| **Path B** (LLM matcher) | 33% | 41% | 41% | **82%** | 20% | 20% | **57%** |
| **A∪B** (B-first union) | 53% | 76% | 88% | **82%** | 33% | 80% | 14% |

(n=51 scored; low/med/high columns are top-1.)

**Headline finding — A and B are complementary, split by query type:**
- **Path B wins the descriptive long-tail** it was built for (low-ambiguity top-1 **82%** vs A's 55%) — rescues exactly what A recall-misses.
- **Path A wins broad nouns + CJK** (high-ambiguity top-1 **80%** vs B's 20%).
- **Path B returns `[]` on CJK / capability-dependent queries** — its catalog is English-descriptions-only, so bare Chinese nouns and IP terms (`chiikawa`) fall through. This is the measured case for enriching the matcher with the capability KB (Option D), *but* — a 6-query try-out showed the naive "dump whole KB into the catalog" only stably recovered 1 query (`植物`), 0 regressions. The promising lever is **per-query targeted KB injection**, not a bigger blob.

## 3. Ontology-enriched gold — per-axis error attribution (new this cycle)

Per the three-tier ontology, every template sits at a cell `Subject × Info-type × Layout`. The enriched gold (`vir_routing_gold_on.json`) attaches that cell to `primary` (= the intended `gold_cell`), every `acceptable` (the multi-valid spread), and every `near_miss` — where each near-miss also records **which axis it diverges from the gold cell on**.

**Why this matters:** it upgrades scoring from flat hit/miss to *"a wrong route is a neighbor cell — but wrong on which axis?"* Across all 76 comparable near-misses:

| Diverges on… | count |
|---|---|
| **layout** | **55** |
| info-type | 28 |
| subject | 21 |
| — *only* layout wrong (right subject + info-type) | **27** |
| — *only* info-type wrong | 3 |
| — *only* subject wrong | 2 |

**Layout is by far the dominant error axis.** This says: routers mostly get the subject right, and the failure is picking the wrong *format*. That directly points the next investment at layout/format signals — and validates the ontology framing empirically.

Example — `marvel mbti character chart 16 types`: near-miss `fandom-character-grid-poster` diverges **only on layout** (right subject `character`+`mbti`, right info-type `collection`, but `grid` vs the gold's `character-card`). Matches the gold's own rationale ("fandom-grid is a 3×3 poster, not a 16-type chart").

**Caveat (honest):** Layout is a *heuristic* from template-id substrings — it is not yet a real queryable tag in the catalog. 13 near-misses came out coordinate-identical to their gold cell (heuristic too coarse to separate). Layout-axis numbers are approximate until Layout becomes a first-class tag.

## 4. The gate — human review of the gold (blocker)

The gold is a Claude draft. Two consequences:

1. **Contamination:** the draft was written *reading the capability KB*, so using it to score a KB-driven router (Option D) would be self-grading — any KB-lift number is untrustworthy until an independent human sign-off.
2. **Known defects to resolve** (surfaced automatically into a decision-only checklist — one yes/no per row, with rationale + evidence inline):

| Category | Rows | What to decide |
|---|---|---|
| Content-gap calls (`acceptable=[]`) | 7 | Real gap, or missed route? (4 conflict with the eval note's `expected=moderate`) |
| Capability overrides | 17 | Gold picked a primary from example evidence over what the note named — justified? |
| Subject-gate drops | 21 | Note named a template; gold demoted it on subject mismatch — correct? |
| Cross-query consistency | 52 | A template rejected here but accepted elsewhere (e.g. `recipe` rejected for `easy weeknight dinners` but accepted for `cuban sandwich`); `吉伊卡哇`→gap but `chiikawa`→acc=3 |

Plus one data quirk the validator flags: `卡通` has 3 acceptable templates but no `primary`.

**Ask for review:** the 4 ⚠️ content-gap conflicts and the 🚩 gap-query consistency rows are the highest-leverage — resolving those unblocks a clean KB-lift measurement.

## 5. Next steps (in order)

1. **Human sign-off on the gold** via the checklist → set `review_status: human-reviewed`. *(Gate — unblocks everything below.)*
2. **Cell-level scoring** wired into `eval_template_routing.cjs`: per-axis accuracy (subject/info-type/layout) + error-attribution histogram + Template-Diversity@K, reading `vir_routing_gold_on.json`.
3. **Fair KB-lift measurement** (Option D, per-query targeted injection) — only meaningful *after* step 1.
4. **Scale the gold** toward the spec's balanced 100→200 and fold in real user queries.
